# План реализации взаимодействия пользователей и управления будильниками (v2)

Этот план описывает этапы внедрения сетевых будильников и системы взаимодействий (удаленный запуск) на основе [документации](docs/groupInteraction.md).

## 1. Модели данных и Сетевой слой (DTO)
- [ ] **Определить DTO для будильников**:
    - `RemoteAlarmDto`: (id, ownerId, title, alarmTime, enabled, repeatPattern, difficultyLevel, isOverslept).
    - `AlarmPermissionDto`: (targetUserId, permissionType, active).
    - `AlarmActionDto`: (alarmId, actorUserId, actionType, status, messageText).
- [ ] **Обновить `ApiService.kt`**:
    - Эндпоинты для личных будильников: `GET /alarms`, `POST /alarms`, `PATCH /alarms/{id}`, `DELETE /alarms/{id}`.
    - Эндпоинты для прав: `POST /alarms/permissions`, `GET /alarms/permissions`.
    - Эндпоинты для действий: `POST /alarms/{id}/trigger`, `GET /alarms/actions/incoming`.

## 2. Логика повторений (Repeat Patterns)
В приложении используется выбор конкретных дней недели.
- [ ] **Маппинг дней недели**:
    - Реализовать конвертер из локального формата (выбранные дни) в паттерны API:
        - Если выбраны все дни -> `DAILY`.
        - Если не выбрано ничего -> `ONCE`.
        - Для конкретных дней -> `MON`, `TUE`, `WED`, `THU`, `FRI`, `SAT`, `SUN`.
    - *Примечание:* Уточнить, поддерживает ли API массив дней или строковый паттерн.

## 3. Управление правами в группах
Больше нет концепции "групповых будильников". Права выдаются на уровне участников.
- [ ] **Автоматическая выдача прав**:
    - При вступлении пользователя в группу (или создании группы) сервер/клиент должен выдать всем участникам право `TRIGGER` на будильники друг друга.
    - Реализовать вызов `POST /alarms/permissions` для всех `memberId` группы.
- [ ] **Ограничение Trigger**:
    - Действие `TRIGGER_NOW` разрешено только в том случае, если текущий статус будильника цели — `isOverslept = true`.

## 4. Слой данных и Репозиторий
- [ ] **Синхронизация состояния**:
    - Добавить `remoteId` в `AlarmItemModel`.
    - При срабатывании будильника локально: если пользователь не отключил его вовремя, отправлять `PATCH /alarms/{id}` с флагом `isOverslept = true`.
- [ ] **Методы взаимодействия**:
    - `getMemberAlarms(userId)`: получение списка будильников участника группы.
    - `remoteTrigger(alarmId, message)`: отправка триггера (только для тех, кто проспал).
    - `checkIncomingActions()`: проверка входящих "пингов" от друзей.

## 5. UI: Взаимодействие в группе
- [ ] **Экран группы / Список участников**:
    - При клике на участника отображать список его активных будильников (полученных через API).
    - Кнопка **"РАЗБУДИТЬ"** отображается и активна **только если** у будильника участника флаг `isOverslept` равен `true`.
- [ ] **Индикация**:
    - Визуально выделять участников, которые сейчас "просыпают" (например, красная обводка или статус "Overslept").

## 6. Обработка входящего триггера
- [ ] **Принудительный запуск**:
    - Реализовать фоновый мониторинг (Polling/FCM) входящих действий.
    - При получении `actionType = TRIGGER_NOW`: немедленно запустить `AlarmActivity` с отображением сообщения от друга.

## 7. Этапы разработки
1. Обновление сетевых моделей (дни недели `MON-SUN`, поле `isOverslept`).
2. Реализация логики автоматической выдачи прав `TRIGGER` при работе с группами.
3. Добавление синхронизации статуса "проспал" (`isOverslept`) с сервером.
4. UI: Интеграция списка будильников участников в экран группы с проверкой условия "проспал".
5. Реализация приема и обработки входящего триггера.
