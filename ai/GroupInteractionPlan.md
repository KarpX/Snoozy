# План реализации взаимодействия с удалённой базой данных

## Обзор

На основе документации GroupInteraction.md, этот план описывает реализацию взаимодействия с удалённой базой данных через REST API для управления будильниками, правами доступа и действиями. План включает сетевой слой, модели данных, репозитории и синхронизацию с локальной базой данных.

## 1. Настройка сетевого слоя

- [ ] **Добавить зависимости**:
  - Retrofit 2 для HTTP-клиента.
  - OkHttp для интерцепторов (аутентификация, логирование).
  - Gson или Moshi для сериализации JSON.
  - Coroutines для асинхронных вызовов.
- [ ] **Настроить Retrofit клиент**:
  - Базовый URL: `/api/v1/alarms`.
  - Интерцепторы: добавление токена аутентификации в заголовки.
  - Таймауты и обработка ошибок сети.
- [ ] **Обработка аутентификации**:
  - Получение и хранение JWT токена.
  - Автоматическое обновление токена при 401 ошибке.

## 2. Модели данных (DTO)

- [ ] **Определить DTO классы на основе API**:
  - `AlarmDto`: (id: Long, ownerId: Long, title: String, alarmTime: String, enabled: Boolean, repeatDays: List<String>, soundName: String?, difficultyLevel: Int?, isOverslept: Boolean, createdAt: String?, updatedAt: String?).
  - `AlarmPermissionDto`: (id: Long?, ownerId: Long?, targetUserId: Long, permissionType: String, active: Boolean, createdAt: String?).
  - `AlarmActionDto`: (id: Long?, alarmId: Long, actorUserId: Long, targetUserId: Long, actionType: String, status: String, executedAt: String?, messageText: String?).
  - `HealthDto`: (status: String, module: String).
- [ ] **Маппинг между DTO и локальными моделями**:
  - Конвертеры для преобразования DTO в Room entities и обратно.
  - Обработка enum значений (repeatDays, permissionType, actionType, status).

## 3. API сервис

- [ ] **Создать интерфейс ApiService**:
  - `GET /api/v1/alarms`: получение своих будильников.
  - `GET /api/v1/alarms/users/{userId}`: получение будильников пользователя.
  - `POST /api/v1/alarms`: создание будильника.
  - `PATCH /api/v1/alarms/{alarmId}`: обновление будильника.
  - `DELETE /api/v1/alarms/{alarmId}`: удаление будильника.
  - `GET /api/v1/alarms/permissions`: получение выданных прав.
  - `POST /api/v1/alarms/permissions`: выдача права.
  - `POST /api/v1/alarms/{alarmId}/trigger`: удалённый запуск.
  - `POST /api/v1/alarms/{alarmId}/enable`: включение будильника.
  - `POST /api/v1/alarms/{alarmId}/disable`: выключение будильника.
  - `GET /api/v1/alarms/actions/incoming`: история входящих действий.
  - `GET /api/v1/alarms/health`: проверка состояния сервиса.
- [ ] **Обработка ответов и ошибок**:
  - Использование sealed classes для Result (Success, Error).
  - Парсинг HTTP кодов ошибок (400, 401, 403, 404).

## 4. Репозитории и слой данных

- [ ] **Создать репозитории**:
  - `AlarmRepository`: управление будильниками (CRUD операции, синхронизация).
  - `PermissionRepository`: управление правами доступа.
  - `ActionRepository`: обработка действий (триггеры, включение/выключение).
- [ ] **Интеграция с локальной базой данных**:
  - Синхронизация данных: при получении от API сохранять в Room.
  - Оффлайн режим: кэширование запросов, синхронизация при подключении.
  - Обработка конфликтов: last-write-wins или manual merge.
- [ ] **Методы репозиториев**:
  - `syncAlarms()`: синхронизация будильников с сервером.
  - `createAlarmRemote(alarm: AlarmDto)`: создание на сервере.
  - `updateAlarmRemote(id: Long, updates: Map<String, Any>)`: обновление на сервере.
  - `deleteAlarmRemote(id: Long)`: удаление на сервере.
  - `grantPermission(targetUserId: Long, type: String)`: выдача права.
  - `triggerAlarm(alarmId: Long, message: String?)`: запуск будильника.

## 5. Синхронизация и оффлайн поддержка

- [ ] **Стратегия синхронизации**:
  - При запуске приложения: отправлять локальные изменения в базу данных. данные загружать из базы данных не нужно.
  - При изменениях: немедленно отправлять на сервер, обновлять локально.
  - Периодическая синхронизация в фоне.
- [ ] **Очереди запросов**:
  - Для оффлайн: сохранять операции в очередь, выполнять при подключении.
  - Обработка конфликтов при синхронизации.
- [ ] **Мониторинг состояния**:
  - Проверка доступности сервера через `/health`.
  - Обработка сетевых ошибок (timeout, no internet).

## 6. Обработка специфики API

- [ ] **RepeatDays**:
  - Маппинг локального формата в List<String> (MON-SUN).
  - Обработка пустого списка как одноразовый будильник.
- [ ] **Права доступа**:
  - Проверка прав перед выполнением действий (TRIGGER, ENABLE_DISABLE).
  - Кэширование прав для быстрого доступа.
- [ ] **Действия**:
  - Обработка входящих триггеров через polling или WebSocket.
  - Отправка уведомлений при получении действия.
